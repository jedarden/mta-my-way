## Health Check Endpoints

### 1. GET /api/health (JSON)
- **File**: `packages/server/src/app.ts:959`
- **Purpose**: Machine-readable health status with detailed subsystem information
- **Response fields**:
  - `status`: "ok" or "degraded"
  - `timestamp`: ISO 8601 timestamp
  - `uptime_seconds`: Server uptime
  - `feeds`: Array of GTFS-RT feed states (per-feed health, circuit-breaker status)
  - `alerts`: Alert polling subsystem status
  - `delayDetector`: Delay detection status
  - `delayPredictor`: Delay predictor status
  - `equipment`: Elevator/escalator outage status
  - `pushSubscriptions`: Count of active push subscriptions
  - `cacheHitRate`: Cache performance metric
  - `memory`: Process memory usage
  - `failingFeedsCount`: Number of feeds failing >5 minutes
- **HTTP status codes**:
  - `200`: Fewer than 3 feeds failing
  - `503`: 3 or more feeds failing
- **Excluded from CSRF**: Line 453 - `/api/health` is in CSRF exclude list
- **Query validation**: Uses `emptyQuerySchema` (rejects all query parameters)

### 2. GET /status (HTML Dashboard)
- **File**: `packages/server/src/app.ts:1063`
- **Purpose**: Human-readable HTML dashboard for system status
- **Response**: Rendered HTML with:
  - Status banner (ok/degraded)
  - System overview (uptime, cache hit rate, subscriptions, failing feeds)
  - Memory usage metrics
  - GTFS-RT feed status table
  - Service alerts status
  - Delay detection/prediction status
- **Cache**: `Cache-Control: public, max-age=30` (line 1162)
- **HTTP status codes**: Same as `/api/health` (200/503 based on feed health)
- **Note**: Not under `/api/*`, so not affected by API middleware (line 446 comment)
