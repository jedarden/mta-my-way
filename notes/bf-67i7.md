## What was implemented

1. **Migration 018** (`018-add-security-persistence.ts`) — creates 11 SQLite tables:
   `security_api_key_registry`, `security_password_reset_tokens`, `security_password_history`,
   `security_password_reset_attempts`, `security_account_lockouts`, `security_rate_limit_bans`,
   `security_trusted_ips`, `security_notification_preferences`, `security_recent_events`,
   `security_notification_history`, `security_notification_templates`

2. **security-db.ts** — write-through SQLite layer; all write functions are no-ops when DB is
   not initialized (tests work without a DB); `setSecurityDb(db)` wires it after migrations

3. **Middleware wiring** (write-through on every mutation):
   - `api-key-management.ts` → `saveApiKey`, `updateApiKeyLastUsed`
   - `password-management.ts` → `savePasswordResetToken`, `appendPasswordHistory`,
     `savePasswordResetAttempt`, `saveAccountLockout`, `markPasswordResetTokenUsed`, etc.
   - `auth-rate-limit.ts` → `saveRateLimitBan`, `saveTrustedIp`, `deleteTrustedIp`, etc.
   - `suspicious-activity-notifications.ts` → `saveSecurityEvent`, `saveNotificationPreferences`,
     `saveNotificationHistoryEntry`, `saveNotificationTemplate`

4. **Startup hydration** (index.ts, after migrations):
   ```ts
   setSecurityDb(pushDb);
   await initApiKeyRegistryFromDb();
   loadRateLimitDataFromDb();
   initPasswordManagementFromDb();
   initNotificationsFromDb();
   ```
